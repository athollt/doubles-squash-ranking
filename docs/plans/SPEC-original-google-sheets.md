# **Specification: BSC Doubles Squash Ladder Google Sheet \+ Apps Script**

## **1\. Purpose**

Build a Google Sheet and Google Apps Script system to maintain a social doubles squash ladder.

The system must:

* Rank individual doubles players, not fixed pairs.  
* Work from normal social play without requiring formal challenge matches.  
* Use one Google Form submission per session.  
* Capture only player names and games won per player.  
* Avoid capturing points, partners, opponents, or exact match structures.  
* Use everyone present in a session as the strength context for scoring.  
* Recalculate the ladder from raw data whenever settings change.  
* Keep raw data clean so the model can be recalibrated later.  
* Keep the spreadsheet private, with a WhatsApp-friendly ranking output.

## **2\. Core Design Principle**

The input must be easy to remember after a session.

A valid session records:

* The players who played.  
* The number of games won by each player.

No other gameplay detail is required.

Example session:

| Player | Games Won |
| :---- | ----: |
| Atholl | 2 |
| Philip | 6 |
| Mike | 2 |
| Mark | 2 |

Total player-wins \= 12\. In doubles, each completed game produces 2 player-wins, so inferred games played \= 12 / 2 \= 6\.

## **4\. Spreadsheet Tabs**

Create the following tabs.

### **4.1 `Players`**

Stores the master player list.

Columns:

| Column | Description |
| :---- | :---- |
| PlayerID | Stable unique ID, e.g. P001 |
| PlayerName | Display name |
| StatusOverride | Blank, Active, Inactive, Removed |
| Notes | Optional |
| CreatedDate | Optional |
| RemovedFlag | TRUE/FALSE or derived from StatusOverride |

Rules:

* Do not delete players.  
* Removed players are hidden from the visible ladder but retained for historical calculations.  
* Player names should be used in the form dropdown.  
* PlayerID should be used internally wherever possible.

### **4.2 `Settings`**

Stores configurable values.

Recommended starting settings:

| Setting | Value |
| :---- | ----: |
| StartingRating | 1000 |
| KFactor | 160 |
| NewPlayerMultiplier | 2.0 |
| NewPlayerSessions | 5 |
| ReturningPlayerMultiplier | 2.0 |
| ReturningPlayerSessions | 5 |
| BaselineSessionGames | 8 |
| MinSessionWeight | 0.60 |
| MaxSessionWeight | 1.25 |
| ActivityBonusPerSession | 2 |
| ActivityBonusWindowDays | 90 |
| ActivityBonusCap | 30 |
| ActiveThresholdDays | 90 |
| LongAbsenceDays | 90 |
| StrengthScale | 400 |

All players start at 1000\. Do not seed players manually.

### **4.3 `Form Responses`**

Raw Google Form output.

Rules:

* This tab is generated by Google Forms.  
* Do not manually edit rows except to delete a bad form submission.  
* If a session is captured incorrectly, delete the raw row and resubmit the whole session.  
* No separate corrections tab is required.

Expected columns:

| Column | Description |
| :---- | :---- |
| Timestamp | Form timestamp |
| Email Address | If available |
| Player 1 | Name |
| Player 1 Wins | Number |
| Player 2 | Name |
| Player 2 Wins | Number |
| ... | Up to Player 8 |
| Notes | Optional |

### **4.4 `Sessions`**

Cleaned session-level table generated by script.

One row per valid or invalid form response.

Columns:

| Column | Description |
| :---- | :---- |
| SessionID | Stable generated ID, e.g. S00001 |
| Timestamp | From form |
| SubmittedBy | Email if available |
| PlayerCount | Count of selected players |
| TotalPlayerWins | Sum of all wins |
| InferredGames | TotalPlayerWins / 2 |
| IsValid | TRUE/FALSE |
| ValidationErrors | Text list of errors |
| Notes | From form |

Validation rules:

A session is invalid if:

* Fewer than 4 players are selected.  
* Any selected player has blank wins.  
* Any selected player has negative wins.  
* Total player-wins is odd.  
* Duplicate players appear in the same session.  
* A selected player does not exist in `Players`.  
* Total player-wins is zero.

Invalid sessions must be excluded from ratings.

### **4.5 `Session_Players`**

Normalised session-player table generated by script.

One row per player per valid session.

Columns:

| Column | Description |
| :---- | :---- |

| PlayerName | Display name |

| InferredGames | Session inferred games |
| ActualShare | Wins / TotalPlayerWins |

### **4.6 `Ratings_Log`**

Chronological rating calculation log generated by script.

One row per player per valid session.

Columns:

| Column | Description |
| :---- | :---- |
| SessionID | Session ID |
| Timestamp | Session timestamp |
| PlayerID | Player ID |
| PlayerName | Display name |
| RatingBefore | Rating before session |

| ExpectedShare | StrengthWeight / SessionStrengthTotal |
| ActualShare | Wins / TotalPlayerWins |
| SessionWeight | Calculated from inferred games |
| Multiplier | New/returning player multiplier or 1 |
| RatingChange | Calculated delta |
| RatingAfter | Rating after session |
| Wins | Games won |
| InferredGames | Session inferred games |
| IsNewOrReturningBoost | TRUE/FALSE |

### **4.7 `Current_Ratings`**

Generated current player summary.

Columns:

| Column | Description |
| :---- | :---- |
| PlayerID | Player ID |
| PlayerName | Display name |
| CurrentRating | Latest rating after all sessions |
| SessionsPlayed | Total valid sessions |
| SessionsLast90Days | Valid sessions in last 90 days |

### **4.8 `Ladder`**

Final ranking table.

Active players are ranked first by LadderScore descending.

Inactive players are shown below active players and sorted by LastPlayedDate descending, so the most recently inactive players appear above players who have not played for a long time.

Removed players are excluded.

Columns:

| :---- | :---- |

## **5\. Rating Algorithm**

The script must rebuild ratings from scratch each time it runs.

### **5.1 Initial Rating**

Every player starts at:

```
1000
```

No initial seeding.

### **5.2 Session Actual Share**

For each valid session:

```
TotalPlayerWins = sum of all player wins
ActualShare = PlayerWins / TotalPlayerWins
InferredGames = TotalPlayerWins / 2
```

### **5.3 Strength Weight**

For each player in the session:

```
StrengthWeight = EXP((RatingBefore - 1000) / StrengthScale)
```

Default:

```
StrengthScale = 400
```

### **5.4 Expected Share**

```
ExpectedShare = PlayerStrengthWeight / SumOfSessionStrengthWeights
```

This allows the session to account for who was present, even though exact partners and opponents are unknown.

### **5.5 Session Weight**

Longer sessions should count more, but not too much more.

```
RawSessionWeight = SQRT(InferredGames / BaselineSessionGames)
SessionWeight = MIN(MaxSessionWeight, MAX(MinSessionWeight, RawSessionWeight))
```

Default values:

```
BaselineSessionGames = 8
MinSessionWeight = 0.60
MaxSessionWeight = 1.25
```

### **5.6 New Player and Returning Player Multiplier**

Each player gets a 2.0 multiplier for their first 5 valid sessions.

```
Multiplier = 2.0 if SessionsPlayedBeforeThisSession < 5
Multiplier = 1.0 otherwise
```

Returning player logic:

If a player has not played for at least `LongAbsenceDays`, apply the same 2.0 multiplier for their next 5 sessions after returning.

Default:

```
LongAbsenceDays = 90
ReturningPlayerMultiplier = 2.0
ReturningPlayerSessions = 5
```

Implementation note: Track a return boost counter during recalculation. When a player returns after a long absence, set `ReturningBoostSessionsRemaining = 5`.

### **5.7 Rating Change**

```
RatingChange = KFactor × SessionWeight × Multiplier × (ActualShare - ExpectedShare)
```

Default:

```
KFactor = 160
```

Then:

```
RatingAfter = RatingBefore + RatingChange
```

### **5.8 Activity Bonus**

Activity bonus rewards regular play but is capped.

```
ActivityBonus = MIN(ActivityBonusCap, SessionsLast90Days × ActivityBonusPerSession)
```

Default:

```
ActivityBonusPerSession = 2
ActivityBonusCap = 30
ActivityBonusWindowDays = 90
```

### **5.9 Ladder Score**

For active players:

```
LadderScore = CurrentRating + ActivityBonus
```

For inactive players, LadderScore is not used for ordering above active players.

## **6\. Active, Inactive, and Removed Logic**

### **6.1 Active**

A player is active if:

* They are not removed.  
* They have played in the last 90 days.

```
DaysSinceLastPlayed <= ActiveThresholdDays
```

Default:

```
ActiveThresholdDays = 90
```

### **6.2 Inactive**

A player is inactive if:

* They are not removed.  
* They have historical results.  
* They have not played in the last 90 days.

Inactive players are sorted below active players by `LastPlayedDate` descending.

### **6.3 Removed**

A player is removed if:

```
StatusOverride = "Removed"
```

Removed players:

* Do not appear on the visible ladder.  
* Remain in the Players table.  
* Remain in historical rating calculations.  
* Must not have their raw historical results deleted.

## **7\. Ladder Sorting**

The final ladder should be sorted in groups:

### **Group 1: Active Players**

Sort by:

```
LadderScore descending
```

### **Group 2: Inactive Players**

Sort by:

```
LastPlayedDate descending
```

### **Group 3: Removed Players**

Exclude from visible ladder.

## **8\. WhatsApp Output**

Create a cell or generated range that produces a simple copy/paste ranking.

Example:

```
BSC Doubles Squash Ladder

1. Philip ↑2
2. Atholl —
3. Mike ↓1
4. Mark ↑1
5. James (P)

Updated after games on 5 Jun
```

Requirements:

* Do not show rating points.  
* Do not show formulas.  
* Do not show inactive players in the default WhatsApp output unless explicitly configured.  
* Show rank movement if possible.  
* Mark provisional players during their first 5 sessions with `(P)` if desired.

## **9\. Apps Script Requirements**

Create an Apps Script project attached to the Google Sheet.

### **9.1 Required Functions**

Implement these functions:

```
onOpen()
```

Adds a custom menu:

```
Squash Ladder
- Recalculate Ladder
- Validate Sessions
- Generate WhatsApp Text
```

```
recalculateLadder()
```

Main function. Rebuilds all derived tabs from raw data.

```
loadSettings()
```

Reads the Settings tab into a config object.

```
loadPlayers()
```

Reads Players tab and maps PlayerName to PlayerID.

```
loadFormResponses()
```

Reads raw form responses.

```
buildSessions()
```

Creates cleaned Sessions rows and validates them.

```
buildSessionPlayers()
```

Creates normalised Session\_Players rows.

```
calculateRatings()
```

Processes valid sessions oldest to newest and creates Ratings\_Log.

```
buildCurrentRatings()
```

Creates Current\_Ratings from latest rating state.

```
buildLadder()
```

Creates final Ladder sorted according to active/inactive/removed rules.

```
buildErrors()
```

Creates Errors tab from invalid sessions.

```
generateWhatsAppText()
```

Creates copy/paste ranking text.

### **9.2 Recalculation Behaviour**

`recalculateLadder()` must:

1. Clear generated tabs:  
     
   * Sessions  
   * Session\_Players  
   * Ratings\_Log  
   * Current\_Ratings  
   * Ladder  
   * Errors

   

2. Read `Settings`.  
     
3. Read `Players`.  
     
4. Read `Form Responses`.  
     
5. Validate and normalise sessions.  
     
6. Exclude invalid sessions from rating calculations.  
     
7. Process valid sessions chronologically.  
     
8. Recalculate all ratings from starting rating 1000\.  
     
9. Build current ratings.  
     
10. Build ladder.  
      
11. Build WhatsApp output.  
      
12. Write all outputs back to their tabs.

### **9.3 Trigger**

Start with manual recalculation via custom menu.

Later, optionally add an installable trigger to run recalculation after form submission.

## **10\. Validation Details**

Each raw form response should be checked for:

* At least 4 players.  
* No duplicate players.  
* All selected players exist in `Players`.  
* Selected players have numeric wins.  
* Wins are zero or positive.  
* Total player-wins is even.  
* Total player-wins is greater than zero.  
* Optional: flag sessions where any individual player has more than 20 wins.

Invalid rows should:

* Appear in `Errors`.  
* Appear in `Sessions` with `IsValid = FALSE`.  
* Be excluded from `Session_Players`, `Ratings_Log`, `Current_Ratings`, and `Ladder`.

## **11\. Data Preservation Rules**

* Do not delete players from `Players`.  
* Do not delete historical results unless a form row was captured incorrectly.  
* No corrections workflow is required.  
* To correct a bad session, delete the raw `Form Responses` row and resubmit the full session.  
* Recalculation must always rebuild from raw form data and settings.

## **12\. Future Web App Compatibility**

Design the sheet as if it may later become a database-backed web app.

Use:

* Stable Player IDs.  
* Stable Session IDs.  
* Append-only raw form rows, except for deleting incorrect captures.  
* Separate Settings.  
* Normalised Session\_Players table.  
* Ratings\_Log as an auditable calculation trail.

Do not hard-code player names in script logic. Use the Players table.

## **13\. Acceptance Criteria**

The implementation is complete when:

1. A Google Form can capture a session with up to 8 players.  
2. Raw form data appears in `Form Responses`.  
3. Running `Recalculate Ladder` creates all derived tabs.  
4. Invalid sessions are flagged and excluded.  
5. All players start at 1000\.  
6. First 5 sessions receive a 2.0 multiplier.  
7. Session score uses actual share vs expected share.  
8. Session strength uses all players present.  
9. Activity bonus is applied and capped.  
10. Inactive players drop below active players.  
11. Removed players are hidden.  
12. The final ladder is a strict ranking.  
13. A WhatsApp-friendly ranking can be copied without rating points.  
14. Changing Settings and rerunning recalculation updates the whole ladder from raw data.
